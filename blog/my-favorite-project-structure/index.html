My favorite project structure

It's been some time that I'm working in the industry and with time comes knowledge but also personal preferences. In this post, I would like to share with you my favorite way to organize a React project. This way of organizing files and folders has been introduced to me almost two years ago by a former colleague Aleksei Ustiuzhanin and scales very well for the kind of project I'm working on. Let's start.

When building an application, we're focusing on providing value to people. Generally we do that by creating a set of "features" that respond to different needs. Let's represent this notion at the file system level level.

.
└── src
    └── features
        ├── FeatureA
        └── ManageInvoices

A feature is flow of one or multiple pages (or one or multiple actions) in order to respond to a particular need. In this sense, I tend to create subfolders for pages and anything that I would eventually need at this point.

.
└── src
    └── features
        ├── FeatureA
        └── ManageInvoices
            └── pages
                ├── AddInvoice.tsx
                └── EditInvoice.tsx

An application is tightly coupled to a specific business.

For instance an application aiming to provide editing tools for invoices does not deal with the same business as an application that manages appointments on agendas.

However, I would like to nuance this notion. "The business", in software engineering, often corresponds to a specific piece of code that does not deal with side effects and only represent business interactions. Hopefully, this blog post is about structuring a project in the frontend and even if people are shifting some logic from the backend to the frontend, we (frontend folks) don't own 100% of the business.

But still, there is a piece of it on the frontend side that is often wrapped in library specific code (hooks or components with React for example).

To avoid confusion with the "real business", I often create a folder called "modules" in which I put everything related to the business interactions plus additional stuff like fetching, computing and eventually displaying information. I also include cross cutting concerns such as authentication, authorization, logging etc...

.
└── src
    ├── features
    │   ├── FeatureA
    │   └── FeatureB
    └── modules
        ├── Authentication
        ├── Billing
        ├── Invoice
        └── User

To prevent chaos from happening, I try to stick as much as possible to the following rules:

  • One module (and its internals) can be used by another module
  • One module can be used in multiple features
  • One feature internals can not be used in another features

The last rule might sound a bit difficult: what about reusable piece of logic / code?

I have a relatively strong opinions on this one.

Answering this question is hard. My perception of this is that what we want to abstract (I mean, making it reusable in different contexts without tightly coupling the thing) does not reside in "avoiding duplication". For instance, abstracting an utility / helper because two components are using something that look that way is not a sufficient argument for me to create an abstraction. One example that I like to provide when talking about abstraction is UI components and layout components.
